2 · Why this stack is materially better for agentic codegen

Stack choice is not aesthetics. The locked stack (Next.js 14 + Supabase Postgres + RLS-first + TypeScript end-to-end) was picked deliberately in stack-decision.md; agentic codegen amplifies every one of those choices. The vendor's pivot to Vue + Laravel + MySQL undoes the same amplifications.

Stack propertyWhy codegen produces fewer landmines
TypeScript end-to-end Hallucinated column names and shape mismatches fail at compile time, not at runtime in a clinic. Single biggest quality lever an agentic dev can hold.
supabase gen types typescript Schema is the source of truth. Types regenerate after every migration. Codegen consumes the generated types and stops inventing fields it has not seen.
RLS policies are pure SQL A policy is 5–15 lines, easy for an LLM to draft, easy for a human to read, fails closed by default. Compare to Laravel policies — 200+ controller files where forgetting $this->authorize() once equals a leak.
Server Components + Server Actions Eliminates a layer (REST API route + client fetch + duplicated validation). Less surface for codegen to misalign client and server.
Single language across surfaces Frontend, server actions, edge functions all in TS. Codegen's context window stays coherent across files; PHP+JS forces two mental models.
Managed Supabase + Vercel/Cloudflare Codegen does not write nginx, supervisor, queue worker daemons, redis tuning, log rotation, php-fpm pool sizing. That work disappears entirely. Saves ~1 hour/week of ops indefinitely.
Drizzle / Supabase-js typed query builders Column references are statically checked. The single most common LLM SQL error — WHERE user_id = … when the column is userId — fails compile.
Supabase Vault + app_secret() helper Crypto key custody is a one-time setup; codegen never sees the key. On Laravel + APP_KEY, codegen will routinely write pgp_sym_encrypt('PHI', env('KEY')) patterns and tempt fate.
RLS attack tests The data-model.md §7.4 adversarial test suite runs against the database, not against the application. Codegen can write 100 controllers; if RLS holds, it does not matter that 3 of them forgot to authz. This is the single biggest reason the spec was written Postgres-first.
The most under-rated point. RLS converts "did I remember to authz this controller?" from a per-route human review (which fatigues a solo dev after week 4) into a single SQL test suite. That is the architecture paying you back exactly when codegen's failure mode bites — 80% review acceptance + 20% rubber-stamping after fatigue sets in.

3 · Where codegen struggles on this stack (honest)

Symmetry: it is not a free lunch. Six places where agentic codegen on this exact stack costs more than naive estimates assume:

  1. RLS debugging tax. "Why is this query returning empty when I'm clearly logged in?" Famous time-eater. When a test fails, codegen's reflex is to broaden the policy; you must enforce the discipline of "narrow first, prove the case, then accept." Budget 1.2–1.5× on RLS-touching work.
  2. Supabase Auth custom_access_token_hook. Documentation thin. Codegen often produces the older raw_user_meta_data pattern that does not work for RLS. You will fight this once, then it sticks. Add 2 days.
  3. Storage RLS is separate from table RLS. Materials, SOAP attachments, exports: storage policies live in their own surface. Codegen will write table policies and forget storage. Add 1–2 days.
  4. pgcrypto + Vault wiring. Codegen reflexively writes pgp_sym_encrypt('val', 'plaintext'). You must enforce app_secret('key_name') via Vault every single time. This is a code-review discipline issue, not a codegen issue.
  5. Edge runtime vs Node runtime. Codegen mixes them. Some libraries (Razorpay SDK, certain crypto helpers) do not run on Edge. You hit this exactly once per surface and it costs an hour each time.
  6. Clinical scoring functions. PHQ-9, GAD-7, PCL-5, OCI-R, MSI-BPD, SCOFF, AUDIT-C all have edge cases (reverse-coded items, "difficulty" question on PHQ-9 not scored, AUDIT-C cutoffs differ by gender). Codegen produces scoring that looks correct and silently miscalculates 1–2 tools. You must hand-verify against published scoring sheets, not against codegen's tests. Codegen tests will pass against codegen's understanding.

7 · Internal audit — where these estimates are wrong

7.1 · Where I am probably under-estimating

  1. RLS debugging tax. Every senior Postgres-on-Supabase dev reports the same time-eater. Estimates assume 1.2× overhead on RLS-touching work; reality may be 1.5×. Add ~3–5 dev-days.
  2. Codegen review fatigue. When you accept 80% of generated code, the 20% you do not catch compounds. After ~6 weeks of solo agentic dev you are rubber-stamping. Mitigation: schedule one "no-codegen review day" per week where you re-read what you accepted last week with fresh eyes. This costs ~10% time and saves the project. Add ~10% across the board.
  3. The two-hat problem. 25h/week assumes you do not lose Wednesdays to clinic emergencies. You will. Apply a 30% calendar buffer to anything quoted. 24 calendar weeks → 31 calendar weeks realistic with clinic load.
  4. Custom_access_token_hook + RLS interaction with Realtime / Edge Functions is documented thinly and codegen will mis-pattern it. Add 2 days.
  5. Supabase Storage signed-URL handling for materials + SOAP attachments has subtle RLS interplay. Storage policies are separate from table policies. Add 1–2 days.
  6. MSG91 template approval cycles are calendar-blocking and Meta is famously inconsistent. Do not gate launch on this — fall back to SMS for V1 launch. Calendar risk, not dev-day risk.
  7. Vault key-rotation playbook. First time you rotate a Vault secret in production with encrypted columns referencing it, you will lose a day. Plan for it before V1, do not discover it after.
  8. The 5-tester study almost always surfaces a routing or wording change. Budget 2 days of revisions, not 0.

7.2 · Where I am probably over-estimating

  1. Intake prototype. You already have intake-spec.md at "engineer can scaffold from this" detail. Codegen could eat this in 4–5 days flat for V1, not 7–10. Bumped because of CF feedback loops; if CF is responsive, lower bound holds. Subtract ~2 days.
  2. Earnings ledger / payout config. The schema in data-model.md is so specific that codegen has very little room to mis-interpret. Could be 2 days, not 3–4. Subtract ~1 day.
  3. Admin portal. With refine.dev or shadcn-admin templates + codegen, this could compress to 5–6 days, not 8–12. Subtract ~3 days if templating.
  4. Production cutover. If staging mirrors prod (which it should), cutover is a half-day, not 3–4. Subtract ~1–2 days.

7.3 · Net of corrections

Optimistic centre with corrections: ~110 → ~115 dev-days (over-estimation savings of ~7 days, under-estimation additions of ~12 days; net +5).

Realistic centre with corrections: ~140 → ~150 dev-days. Calendar with two-hat 30% buffer: ~28–31 calendar weeks. ~7 calendar months from kickoff to launch.

This tracks Saday_Action_Plan_by_Claude.md's 24-week Phase 0–4 plan, which Akash drafted before assuming agentic dev. The Action Plan's calendar was not bottlenecked on typing speed in the first place. It was bottlenecked on CF time, clinical sign-off, legal review, and 5-tester studies — none of which codegen accelerates. Agentic dev compresses build, not project.

Cumulative scoring

Solo path scores best when

  • Cash is the binding constraint
  • You want maximum codebase ownership
  • Bus-factor concerns can be deferred
  • Clinic load is genuinely 50%-of-week or less for 7 months
  • You expect to keep building this product yourself for 1–2 years

Hybrid path scores best when

  • Speed-to-launch matters (14–16 weeks)
  • You can identify a competent contractor (₹1.5–2.5k/hour senior, Indian Next.js+TS pool, Postgres-fluent)
  • Safety-critical surfaces are non-negotiable in your hands
  • Cash budget is mid (₹6–11L year 1 acceptable)
  • You want a clear handoff boundary (you do schema/RLS/audit/intake/scoring; contractor does dashboards/payments/video/notifications)
My honest read. Solo agentic is a credible and respectable path on this stack — more credible than on any other stack you could have picked. But the project is gated by CF time and external dependencies, not coding speed, so the gain over hybrid is smaller than it looks. The hybrid path lands ~10–12 weeks earlier with the same correctness guarantees on the load-bearing surfaces, and a ₹2–3L contractor spend is <1.5× a single month of clinic revenue equivalence. Hybrid is the rational default. Solo is the right call if cash is the binding constraint or if owning every line is a hard requirement for you.

11 · Open questions for Akash to answer before locking a path

  1. Honest 8-week-ahead clinic load forecast. Will you genuinely have 25h/week for 7 months, or is it 15h/week with growth into 25? The latter pushes solo to ~10 calendar months.
  2. Contractor pool reality. Do you have a known Next.js+TS+Postgres senior in your network, or do you need to recruit cold? Recruiting cold adds 4–6 weeks before contractor work starts and is a hidden risk.
  3. What is the launch trigger? First 10 patients? First MHP onboarded? First DPDP-compliant audit-ready checkpoint? The trigger changes which surfaces are V1 versus deferred.
  4. Hindi at launch — required, or fast-follow? If fast-follow, you save 1.5–2 calendar weeks of CF blocking time on launch.
  5. Are you willing to ship V1 EN-only with HI 4 weeks later? Decoupling translation from build calendar is a strong simplification.
  6. Is admin portal a V1 must-have or V1.1 acceptable? If admin can be SQL + a Notion runbook for 60 days post-launch, you save 2 weeks of dev. Patients and MHPs cannot wait; admin can.
  7. Solo dev review cadence. Are you willing to enforce a weekly fresh-eyes review day at the cost of ~10% throughput? If not, expect the failure mode to bite around week 14.
